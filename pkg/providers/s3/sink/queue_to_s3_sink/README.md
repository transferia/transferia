# AsyncSink - Компонент загрузки данных из очередей в S3

## Основные компоненты

[`SnapshotWriter`](../snapshot_writer.go) — используется для записи данных, подробнее [здесь](../README_SNAPSHOT_SINK.md)

[`Rotator`](rotator.go) — хранит настройки ротации, дает сигнал, когда `AsyncSink` должен завершить запись в старый файл и начать записывать данные в следующий

[`Partitioner`](partitioner.go) — хранит в себе настройки, связанные с формированием пути файла в бакете, ответственен за формирование этих путей

## Принцип работы
# Расчитываем на то, что данные идут из одной партиции

1. **Начало работы** - 
1. 1. Получаем первый батч с изменениями 
1. 2. Обновляем статус ротатора
1. 3. Формируем путь к файлу
1. 4. Инициализируем писателя
1. 5. Отправляем данные на запись

2. **Обработка батча, не требующая роацию**
2. 1. Данные отправляются на обработку и запись `SnapshotWriter`, процесс завершается
2. 2. Сохраняем список записанных оффсетов

3. **Обработка батча, требующая роацию** - 
3. 1. Находим в батче первый `ChangeItem`, который в соответствии с настройками ротатора должен быть помещен в новый файл s3
3. 2. Бьем батч на две части, `ChangeItem` из предыдущего пункта становится первым во второй части
3. 3. Первую часть батча обрабатываем согласно пункту 2 (*Обработка батча, не требующая роацию*)
3. 4. Коммитим писателя, успешный    коммит гарантирует, что новый файл успешно появился в S3
3. 5. Отправляем список оффсетов, попавших в только что записанный файл, на источник, очищаем список на стороне приемника
3. 6. Проводим инициализацию, аналогичную пункту 1 (*Начало работы*)
3. 7. Вторую часть батча повторно отправляем на проверку необходимости ротации
