## Как устроено шардирование репликации в этом коде

Шардирование здесь — это не разбиение данных на партиции, а паттерн конкурирующих потребителей (competing consumers) на одной SQS-очереди. Несколько воркеров читают из одной очереди, а SQS сама раздаёт сообщения
— кто первый poll сделал, тот и получил.

Механика на стороне SQS:

У каждого сообщения в очереди нет флага "commit", но есть механика из двух операций — получить и удалить:

1. ReceiveMessage — любой клиент запрашивает до 10 сообщений и сам задаёт Visibility Timeout (в коде — 600 секунд). На этот период сообщения становятся невидимыми для всех остальных клиентов. Никакой серверной
   логики распределения, кроме случайной выдачи тому, кто polling делает.
2. DeleteMessage (аналог commit) — клиент явно удаляет сообщение из очереди. После удаления оно исчезает навсегда и больше никому не достанется.
3. Если клиент не удалил сообщение за время Visibility Timeout — оно автоматически снова становится видимым, и любой другой клиент может его получить. Это защита от падения воркера.

Как это используется в коде (object_fetcher_sqs.go):

- fetchMessages (строка 83) — получает батч сообщений с Visibility Timeout 10 минут
- visibilityHeartbeat (строка 184) — фоновая горутина, которая каждые 5 минут продлевает таймаут ещё на 10 минут для сообщений, которые ещё обрабатываются. Нужно, чтобы длительная обработка файла не привела к
  ложному "возврату" сообщения.
- Commit (строка 278) — удаляет сообщение из очереди после успешной обработки файла.

Итог: воркеры соревнуются за сообщения, SQS раздаёт их произвольно, Visibility Timeout защищает от одновременной обработки одного файла двумя воркерами, а heartbeat не даёт таймауту истечь, пока файл реально
обрабатывается. Никаких coordinator'ов, никаких партиций — только очередь, таймаут видимости и явное удаление.
